20  @section STLOverview Overview
21 
22 Sidekiq was developed under the assumption that there would always be either a
23 PCIe or USB interface available to connect the host system and the card itself.
24 This has worked out reasonably well up until now, with a common libsidekiq
25 library capable of supporting either of these interfaces. Most developers will
26 only need to use the "stock" transport layers. The software/FPGA architecture
27 for using PCIe as the transport interface is shown below.
28 
29 @image html sidekiq_software_fpga_block_diagram.png
30 @image latex sidekiq_software_fpga_block_diagram.eps "Software / FPGA Architecture" width=10cm
31 
32 However, for some applications, it is necessary to provide a custom transport
33 interface between libsidekiq and the FPGA. In this case, the architecture to
34 support this would need to have a custom software layer at the bottom of the
35 software stack, as well as a custom FPGA interface. Hosting a Sidekiq card in
36 custom hardware may benefit from a custom transport layer. This alternate
37 architecture with a custom transport is shown below.
38 
39 @image html sidekiq_software_fpga_alternate_transport_block_diagram.png
40 @image latex sidekiq_software_fpga_alternate_transport_block_diagram.eps "Software / FPGA Architecture - Custom Transport" width=10cm

42 @section CTInterface Custom Transport Interface
43 
44 Support for a custom transport implementation is available in the latest
45 libsidekiq release starting with v3.2.0 and necessitates the development of
46 three new software/FPGA components. This is detailed in the sections below.
47 
48 - @b libsidekiq: This is the primary library that supports the use of a separate
49  external transport implementation. The existing PCIe and USB transport layers
50  used by libsidekiq are already bundled with libsidekiq and are available for
51  use by applications. Applications that wish to use a custom transport
52  implementation may enhance their application by registering the custom
53  transport implementation (skiq_register_custom_transport()) and initializing
54  libsidekiq to use that custom transport (skiq_init_xport()).
55 
56  - <tt>int32_t skiq_register_custom_transport(skiq_xport_card_functions_t *
57  functions)</tt> - This function performs registration of all the required card
58  function pointers so that libsidekiq knows which functions should be called in
59  the custom transport implementation for the required transport operations.
60  Card functions (probe, init, exit) are registered with this function. Probe
61  and init functions can register functions related to FPGA, RX, and TX
62  functionality depending on a custom transport's capability. This includes
63  functions for reading/writing FPGA registers, starting/stopping streaming,
64  receiving contiguous blocks of samples, and flushing the transport. See the
65  @ref CTImplementationGuide "Sidekiq Custom Transport Implementation Guide"
66  section in Sidekiq API documentation for more details.
67 
68  - <tt>int32_t skiq_init ( skiq_xport_type_t type,
69  skiq_xport_init_level_t level, uint8_t * p_card_nums, uint8_t num_cards )</tt>
70  - This function is used to initialize the library and cards. If using the
71  custom transport implementation, the transport functions must be registered
72  prior to calling this API. The specified level may be '
73  @ref skiq_xport_init_level_basic "basic"' where only FPGA related functions
74  are registered, or '@ref skiq_xport_init_level_full "full"'where RX / TX
75  streaming related functions are registered. See the Sidekiq API
76  documentation for more details on this function and related enumerations.
77 
78 - <b>Custom SW "Driver"</b>: This is the Linux kernel space driver that may be
79  required by the custom transport layer to support both register and streaming
80  operations with hardware. The assumption here is that the custom transport
81  implementation calls this software interface to provide the physical transport
82  between the CPU and the FPGA.
83 
84 - <b>Custom FPGA Interface</b>: This is the FPGA block that manages both the
85  register and streaming interfaces on the FPGA. The assumption here is that
86  the vast majority of the stock Sidekiq FPGA reference design will be
87  preserved, replacing only the PCIe/DMA interface currently provided by
88  Northwest Logic with the custom transport interface.
89 
90  @section CTImplementationGuide Custom Transport Implementation Guide
91 
92  @subsection Overview Overview
93 
94 The current implementation of custom FPGA transport support has the ability to
95 perform register transactions as well as receive sample streaming. While the
96 transmit sample streaming hooks are in place, a custom transport implementation
97 may leave them at their defaults. This release is based on libsidekiq v3.1.0
98 and adds the custom FPGA transport support and is planned to be released as
99 libsidekiq v3.2.0.
100 
101  @subsection TransportImplementation Transport Implementation
102 
103 In the SDK's top-level directory there are two new directories that have example
104 implementations. The first, @b custom_xport_bare, is a simple stub example
105 where all functions print their parameters and return success. This example can
106 be used as a basis for new work. The second example, @b custom_xport_example,
107 is the PCIe transport layer implementation and can be compiled and referenced as
108 a complete example.
109 
110  @subsection Compiling Compiling the Transport Implementation
111 
112 -# Change directory to either @b custom_xport_bare or @b custom_xport_example
113 -# Type <tt>make BUILD_CONFIG=x86_64.gcc</tt>
114 
115 This builds the custom transport and some example test applications. A test
116 application's object file is linked with the custom transport object(s) and
117 libsidekiq's static library. The example test applications are the same as
118 their test_apps counterpart with the exception of calling
119 skiq_register_custom_transport(), and specifying
120 ::skiq_xport_type_custom in calls to skiq_init().

122  @subsection API API
123 
124 Below is a summary of the function sets that should be supported by the transport library.
125 
126  @subsection CardFunctions Card Functions
127 
128 All three card functions are required for a custom transport implementation.
129 Pointers to the functions are collected into a ::skiq_xport_card_functions_t
130 struct and passed to skiq_register_custom_transport() function before calling
131 skiq_init_xport().
132 
133 - <tt>int32_t (*card_probe)( uint64_t *p_uid_list, uint8_t *p_num_cards );</tt>
134 - <tt>int32_t (*card_init)( skiq_xport_init_level_t level, uint64_t xport_uid );</tt>
135 - <tt>int32_t (*card_exit)( skiq_xport_init_level_t level, uint64_t xport_uid );</tt>
136 
137 The @ref card_probe and @ref card_init are responsible for further registering
138 the remaining transport function sets (FPGA, RX, and TX) based on the caller's
139 ::skiq_xport_init_level_t request and the card's capabilities. Refer to the
140 @b custom_xport_bare example.
141 
142  @subsection FPGAFunctions FPGA Functions
143 
144 The FPGA functions implement transporting requests to read / write registers as
145 well as bringing up / down the transport link to the FPGA (for reprogramming).
146 Pointers to the functions are collected into a ::skiq_xport_fpga_functions_t
147 struct and registered on a per-card basis by calling
148 xport_register_fpga_functions().
149 
150 - <tt>int32_t (*fpga_reg_read)( uint64_t xport_uid, uint32_t addr, uint32_t* p_data );</tt>
151 - <tt>int32_t (*fpga_reg_write)( uint64_t xport_uid, uint32_t addr, uint32_t data );</tt>
152 - <tt>int32_t (*fpga_down)( uint64_t xport_uid );</tt>
153 - <tt>int32_t (*fpga_up)( uint64_t xport_uid );</tt>
154 
155  @subsection RXFunctions RX Functions
156 
157 The RX functions implement preparing the transport link for starting, stopping,
158 pausing, and resuming receive sample data streaming as well as flushing buffered
159 receive data and transporting receive sample data. Pointers to the functions
160 are collected into a ::skiq_xport_rx_functions_t struct and registered on a
161 per-card basis by calling xport_register_rx_functions().
162 
163 - <tt>int32_t (*rx_start_streaming)( uint64_t xport_uid, skiq_rx_hdl_t hdl );</tt>
164 - <tt>int32_t (*rx_stop_streaming)( uint64_t xport_uid, skiq_rx_hdl_t hdl );</tt>
165 - <tt>int32_t (*rx_pause_streaming)( uint64_t xport_uid );</tt>
166 - <tt>int32_t (*rx_resume_streaming)( uint64_t xport_uid );</tt>
167 - <tt>int32_t (*rx_flush)( uint64_t xport_uid );</tt>
168 - <tt>int32_t (*rx_receive)( uint64_t xport_uid, uint8_t **pp_data, uint32_t *p_data_len );</tt>
169 
170  @subsection TXFunctions TX Functions
171 
172 The TX functions implement preparing the transport link for starting and
173 stopping transmit sample data as well as transporting transmit sample data.
174 Pointers to the functions are collected into a ::skiq_xport_tx_functions_t
175 struct and registered on a per-card basis by calling
176 xport_register_tx_functions().
177 
178 - <tt>int32_t (*tx_initialize)( uint64_t xport_uid, skiq_tx_transfer_mode_t tx_transfer_mode, uint32_t num_bytes_to_send, uint8_t num_send_threads, void (*tx_complete_cb)(int32_t status, uint32_t *p_data) );</tt>
179 - <tt>int32_t (*tx_start_streaming)( uint64_t xport_uid, skiq_tx_hdl_t hdl );</tt>
180 - <tt>int32_t (*tx_pre_stop_streaming)( uint64_t xport_uid, skiq_tx_hdl_t hdl );</tt>
181 - <tt>int32_t (*tx_stop_streaming)( uint64_t xport_uid, skiq_tx_hdl_t hdl );</tt>
182 - <tt>int32_t (*tx_transmit)( uint64_t xport_uid, skiq_tx_hdl_t hdl, int32_t *p_samples, void *p_private );</tt>
183 
184 @note It is NOT recommended to use xport_register_fpga_functions(),
185 xport_register_rx_functions(), xport_register_tx_functions(),
186 xport_unregister_fpga_functions(), xport_unregister_rx_functions(), or
187 xport_unregister_tx_functions() from functions other than the @ref card_init and
188 @ref card_exit implementations.
189
